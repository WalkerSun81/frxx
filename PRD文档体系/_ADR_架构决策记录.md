# 架构决策记录（ADR）

> **用途**：记录重大架构和技术决策的背景、方案对比、选型理由  
> **格式**：每个 ADR 包含「背景 → 方案对比 → 决策 → 后果」

---

## ADR-001：纯服务端运算架构

| 元数据 | 值 |
|:---|:---|
| **决策 ID** | ADR-001 |
| **日期** | 2026-05-12 |
| **状态** | 已采纳 |
| **影响范围** | 全系统 |

### 背景

传统客户端游戏中，部分游戏逻辑（伤害计算、掉落判定）在客户端执行，容易被修改内存、Hook 函数等方式作弊。

### 方案对比

| 方案 | 优点 | 缺点 |
|:---|:---|:---|
| **纯服务端运算**（选） | 数据不可篡改，唯一真相源 | 增加服务器负载和网络通信 |
| **客户端运算+服务端校验** | 降低延迟，减轻服务器压力 | 校验逻辑复杂，仍有作弊空间 |
| **混合运算** | 灵活性高 | 维护两套计算逻辑，一致性风险 |

### 决策

所有数值计算（伤害公式、经验收益、掉落概率、境界突破成功率等）完全在服务端完成。客户端仅负责 UI 渲染和数据展示。

### 后果

- 前端变得轻薄，但没有离线玩法的可能性
- 服务端需要处理所有计算，对性能要求更高
- 每个操作都需要网络往返，用户体验受延迟影响

## ADR-002：Node.js 后端选型

| 元数据 | 值 |
|:---|:---|
| **决策 ID** | ADR-002 |
| **日期** | 2026-05-12（V2.1 更新） |
| **状态** | 已采纳 |
| **影响范围** | 服务端技术栈 |

### 背景

最初技术架构文档列出了 Go 和 Java 两个选项。V2.1 明确为 HTML5 + Node.js。

### 方案对比

| 方案 | 优点 | 缺点 |
|:---|:---|:---|
| **Node.js**（选） | 前后端统一语言、轻量、npm 生态丰富、开发效率高 | 不适合 CPU 密集型计算 |
| **Go** | 高并发性能优异、编译部署简单 | 团队需额外学习成本 |
| **Java** | 生态成熟、中间件丰富 | 启动慢、配置复杂、开发效率低 |

### 决策

采用 Node.js 18+，Express/Koa 提供 RESTful API，ws/Socket.IO 提供 WebSocket 服务，SQLite（开发）/ MySQL（生产）作为持久化存储。

### 后果

- 前后端共用 TypeScript/JavaScript，降低上下文切换成本
- 对于高并发场景需要做好异步处理和限流
- 后续 CPU 密集型计算（如排行榜运算）可抽离为独立服务

## ADR-003：JSON 作为通信格式

| 元数据 | 值 |
|:---|:---|
| **决策 ID** | ADR-003 |
| **日期** | 2026-05-12 |
| **状态** | 已采纳 |
| **影响范围** | 前后端通信协议 |

### 背景

前后端通信需要确定数据序列化格式。

### 方案对比

| 方案 | 优点 | 缺点 |
|:---|:---|:---|
| **JSON**（选） | 可读性强、调试方便、跨语言友好 | 数据包体积较大 |
| **Protobuf** | 体积小、解析快、强类型 | 需要 schema 管理、调试困难 |
| **MessagePack** | 体积小、类 JSON | 工具链不成熟 |

### 决策

默认使用 JSON，性能敏感场景（核心战斗流程）可协商切换到 Protobuf。

### 后果

- 降低了开发调试难度，网络抓包即可查看完整数据
- 后期如需切换 Protobuf，需要建立 schema 管理和编译流程

## ADR-004：双通道通信架构（HTTPS + WebSocket）

| 元数据 | 值 |
|:---|:---|
| **决策 ID** | ADR-004 |
| **日期** | 2026-05-12 |
| **状态** | 已采纳 |
| **影响范围** | 前后端通信架构 |

### 背景

不同业务场景对通信模式的要求不同：查询类请求适合无状态 REST，实时推送需要长连接。

### 决策

- **HTTPS**（短连接）：登录认证、角色信息拉取、排行榜查询、商店操作等一次性查询场景
- **WebSocket**（长连接）：战斗过程、挂机状态、属性变化推送、服务器主动通知等实时场景

### 后果

- 客户端需要维护两种连接状态
- WebSocket 断线重连逻辑需要额外实现
- HTTPS 接口便于对接 CDN 缓存

## ADR-005：请求签名 + seq 防重放安全机制

| 元数据 | 值 |
|:---|:---|
| **决策 ID** | ADR-005 |
| **日期** | 2026-05-12 |
| **状态** | 已采纳 |
| **影响范围** | 全系统安全 |

### 背景

由于客户端完全不可信（纯前端 HTML5），需要防止请求篡改和重放攻击。

### 决策

- 签名算法：`MD5(token + seq + JSON(data) + secret_key)`
- seq 递增机制：客户端每次请求 seq 必须递增，服务端记录每个玩家最新 seq，丢弃不递增的请求
- secret_key 存储于服务端，不暴露给客户端

### 后果

- 增加了请求构造的复杂度（每次需计算签名）
- seq 接近 BIGINT 上限时需重新建立连接
- 有效防止抓包篡改和重放攻击

## ADR-006：灵根永久锁定设计

| 元数据 | 值 |
|:---|:---|
| **决策 ID** | ADR-006 |
| **日期** | 2026-05-12 |
| **状态** | 已采纳 |
| **影响范围** | 角色创建、灵根系统 |

### 背景

灵根决定角色的修炼方向和玩法倾向，需要平衡「刷初始」留存机制和「角色不可逆成长」的严肃性。

### 决策

- 灵根在角色创建时随机生成，确认后永久锁定
- 创建阶段提供免费无限重置
- 创建后不可通过任何方式修改灵根

### 后果

- 刷初始阶段成为重要留存钩子（平均 5-10 次重置）
- 玩家决策成本高，需要清晰展示灵根效果
- 降低后期运营中"洗灵根"相关的商业化可能性
