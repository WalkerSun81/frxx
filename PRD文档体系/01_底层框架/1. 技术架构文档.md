## 1.1 架构设计原则

### 核心原则

#### 1. 前端只做展示，所有数据运算和保存由服务端进行

这是本游戏架构的最核心原则。在传统的客户端游戏中，部分游戏逻辑（如伤害计算、掉落判定）会放在客户端执行，然后仅将结果提交给服务端。然而，这种做法在手游环境中极易被破解和作弊（通过修改内存、Hook 函数等方式伪造客户端逻辑）。因此，**凡人修仙录**采用纯服务端运算架构：

- 所有数值计算（伤害公式、经验收益、掉落概率、境界突破成功率等）完全在服务端完成
- 客户端仅负责接收服务端下发的数据并进行 UI 渲染
- 客户端不对任何关键数值进行本地计算或缓存
- 客户端发送的任何操作请求，服务端均需独立校验合法性后方可执行

这种设计虽然会增加服务器计算负载和网络通信量，但换来了极高的安全性和数据一致性保障——即无论客户端如何修改，服务端始终是"唯一真相源"（Single Source of Truth）。

#### 2. 服务端采用无状态设计，支持水平扩展

游戏服务器需要承载大量并发玩家的请求，同时需要应对运营活动带来的流量峰值。为保证系统的弹性伸缩能力，服务端整体采用**无状态设计**：

- **无状态 HTTP 服务**：所有 HTTP 接口（登录、查询、数据拉取等）通过网关层分发到任意游戏服务器节点，节点自身不存储会话状态
- **会话外置**：玩家登录后的认证 Token 和会话数据存储在 Redis 中，任何节点均可通过 Token 独立完成身份校验
- **水平扩展能力**：新增服务器节点只需注册到网关的负载均衡池中，无需对已有系统做任何修改
- **网关层负责路由**：玩家的长连接（WebSocket）通过一致性哈希或最小连接数策略分配到对应节点，断线重连时优先路由到同一节点

无状态设计的优势在于：单节点故障不影响整体服务，新增节点可即时分担压力，运维团队可以在高峰期灵活调整实例数量。

#### 3. 数据实时同步，支持跨设备登录

现代手游玩家习惯于在手机、平板等不同设备间切换游戏进度。本架构要求：

- 所有玩家操作实时同步到服务端持久化存储
- 玩家在任一设备登录后，可以立即看到最新的游戏数据
- 同一账号同时只允许一个设备处于在线状态，新设备登录时强制旧设备下线
- 断线重连时，服务端主动推送断线期间的关键事件（如境界突破成功、系统邮件等）至客户端
- 关键资源变化（灵石、修为、装备强化等）通过 WebSocket 实时推送，确保 UI 数据始终最新

| 层级 | 技术选型 | 主要职责 |

| **数据层** | Redis + MySQL + OSS | 缓存加速、会话管理、持久化存储、文件资源管理 |
| **定时任务层** | Go Cron / XXL-Job | 离线收益结算、每日数据重置、排行榜刷新、数据统计 |

## 1.3 后端控制台评估

#### 是否需要后端控制台

**结论：需要。** 理由如下：

| 维度 | 说明 |
|:---|:---|
| **数据安全** | 纯前端HTML方案下，游戏数据需存储在服务端，防止本地篡改 |
| **跨设备同步** | 玩家可能在不同设备/浏览器打开游戏，需服务端统一存储 |
| **数值运算** | 所有核心数值计算（战斗伤害、突破成功率、掉落概率）必须在服务端完成 |
| **热更新配置** | 游戏配置表（境界数值、装备属性等）需服务端统一管理，支持热更新 |
| **反作弊** | 前端完全不可信，服务端是唯一真相源 |

#### 后端控制台技术选型

| 组件 | 选型 | 说明 |
|:---|:---|:---|
| **运行时** | Node.js 18+ | 轻量、生态丰富、前后端统一语言 |
| **Web框架** | Express / Koa | 提供RESTful API和WebSocket服务 |
| **数据库** | SQLite (开发) / MySQL (生产) | 开发阶段零配置，生产环境可平滑迁移 |
| **实时通信** | ws / Socket.IO | 支持WebSocket推送（属性变化、挂机结算等） |
| **管理后台** | 内置Web管理面板 | 玩家管理、数据统计、配置热更新 |
| **部署** | Docker单容器 | 一键部署，降低运维复杂度 |

#### 前后端交互模式

```
┌──────────────────┐         ┌──────────────────┐
│  HTML5 前端       │         │  Node.js 后端     │
│  (浏览器直接打开)  │◄───────►│  (控制台服务)      │
│                  │  HTTP   │                  │
│  纯展示 + 交互    │  WS     │  数值运算 + 存储    │
└──────────────────┘         └──────────────────┘
```

**前端特点**：
- 单个HTML文件，双击即可在浏览器中运行
- 无需构建工具、无需安装依赖
- 通过fetch API与后端通信
- 所有游戏逻辑由后端API驱动

## 1.4 前端职责（纯展示）

前端在整体架构中扮演**纯展示层**的角色，不包含任何游戏核心逻辑。其具体职责范围如下：

### 1.4.1 UI 渲染和动画表现

- 负责所有游戏界面的绘制，包括主界面、背包、角色详情、战斗场景、地图选择等
- 播放服务端下发动画指令（如攻击特效、境界突破光效、装备强化闪光等）
- 管理 UI 层级和交互状态（弹窗、提示、遮罩等）
- 处理屏幕适配和分辨率兼容

### 1.4.2 玩家输入捕获和转发

- 捕获玩家的所有操作（点击、滑动、输入等）
- 将玩家操作按照协议格式封装为请求，发送给网关层
- **不验证**操作的合法性（该工作由服务端完成）
- 处理操作冷却、按钮禁用状态等 UI 层面的简单交互逻辑

### 1.4.3 本地资源加载和缓存

- 管理前端静态资源（CSS/JS/图片等内联或CDN加载）
- 按需加载远程配置和资源
- 通过API获取最新配置，无需本地版本管理
- 管理本地存档（仅用于快速恢复，核心数据以服务端为准）

### 1.4.4 数据展示（基于服务端下发数据）

- 将服务端返回的数据渲染到对应 UI 组件上
- 处理数据的展示格式化（如灵石数字加逗号、境界名称翻译等）
- 管理前端状态的展示逻辑（如加载进度、错误提示、空状态等）
- 根据服务端下发的推送消息实时刷新 UI 数据

### 1.4.5 前端明确不包含的内容

以下逻辑**严格禁止**在前端实现：

- 伤害数值计算
- 掉落概率判定
- 境界突破成功率计算
- 装备强化成功率计算
- 任何依赖随机数的玩法逻辑
- 玩家数据的本地修改和保存

> **设计意图**：将所有核心逻辑收敛到服务端，可以彻底杜绝内存修改、函数 Hook 等客户端作弊手段。即使玩家使用抓包工具修改了请求参数，服务端也会独立校验并拒绝非法操作。

## 1.5 后端职责（核心运算）

后端作为整个游戏的核心引擎，承担所有游戏逻辑和数据管理的职责。

### 1.5.1 所有数值计算

服务端是**唯一的数值运算中心**。具体包括：

| 计算类别 | 具体内容 |
|:---|:---|
| **战斗伤害** | 普攻伤害、技能伤害、暴击伤害、会心伤害，最终伤害 = (攻击方攻击力 - 防御方防御力) x 属性克制系数 x 技能倍率 x 随机波动 |
| **境界突破** | 突破成功率 = f(当前境界、消耗道具数量、命格加成、保护道具)，突破后属性增长率 |
| **收益计算** | 挂机收益 = 基础收益 x 境界加成 x 灵根加成 x 命格加成 x 时长 |
| **强化成功率** | 装备强化成功率 = f(当前强化等级、装备品质、保护道具) |
| **掉落判定** | BOSS 掉落表、随机数生成（服务端 RNG）、数量计算 |
| **属性加成** | 装备属性加成、宝石属性加成、命格属性加成、灵根属性加成的最终叠加计算 |

### 1.5.2 数据持久化存储

- 所有玩家数据（角色、装备、资源、背包等）实时写入 MySQL
- 热点数据（在线状态、最近操作）在 Redis 中缓存，加速读写
- 定时任务将 Redis 中的脏数据刷回 MySQL
- 关键操作（如大额交易、装备强化）采用事务保证原子性

### 1.5.3 业务逻辑处理

- 账号注册、登录、Token 颁发与校验
- 角色创建、删除、数据初始化
- 战斗流程管理（进入战斗、回合处理、战斗结算）
- 装备的穿戴/卸下/强化/精炼/继承逻辑
- 宝石的镶嵌/拆卸/合成逻辑
- 命格天赋的随机/重置/激活逻辑
- 挂机流程的开启/结算/离线补偿逻辑
- 邮件的发送/领取/过期清理逻辑
- 生活技能的升级/采集/制作逻辑

### 1.5.4 安全校验（防止作弊）

服务端必须对所有客户端请求进行严格的安全校验，这是架构安全的最后一道防线：

- **签名校验**：每个请求携带 `sign` 字段，服务端用相同算法（`MD5(token + seq + data)`）重新计算并比对，防止请求被篡改
- **参数校验**：每个接口参数的类型、范围、枚举值进行严格校验，拒绝非法参数
- **频率限制**：对关键操作（如挑战关卡、装备强化）设置操作频率限制，防止脚本刷资源
- **业务规则校验**：境界未达标不可突破、资源不足不可购买、背包已满不可拾取等
- **异常检测**：监控同一账号的异常操作模式（如同一秒内多次强化），触发风控告警

### 1.5.5 定时任务执行

- **每日重置**：每日 00:00 重置副本挑战次数、领取日常奖励、刷新商店等
- **离线收益计算**：玩家离线期间（最长 N 小时）的挂机收益计算
- **排行榜更新**：定时刷新战力榜、境界榜、财富榜等排行榜数据
- **数据统计**：每日/每周/每月的核心运营数据统计（DAU、付费率、留存等）
- **邮件清理**：自动清理过期邮件及附件
- **存档备份**：每日凌晨对数据库进行全量备份

### 1.5.6 排行榜计算

- 支持多维度排行榜：境界榜、战力榜、财富榜、成就榜等
- 排行榜数据存储在 Redis Sorted Set 中，支持高效的范围查询
- 定时（如每 5 分钟）更新排行榜数据，避免频繁计算
- 排行榜变动时主动推送玩家排名变化

## 1.6 数据流向图

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│   玩家操作    │ ──▶ │  前端捕获    │ ──▶ │  发送请求    │ ──▶ │  服务端校验   │ ──▶ │  逻辑运算    │
│              │     │ (UI 交互)    │     │  (HTTP/WS)   │     │  (签名/参数)  │     │  (数值计算)  │
└──────────────┘     └──────────────┘     └──────────────┘     └──────────────┘     └──────┬───────┘
                                                                                              │
┌──────────────┐     ┌──────────────┐     ┌──────────────┐     ┌──────────────┐              │
│  前端展示    │ ◀── │  推送结果    │ ◀── │  数据存储    │ ◀── │  结果封装    │ ◀─────────────┘
│  (UI 更新)   │     │  (WS Push)   │     │  (MySQL)     │     │  (JSON)      │
└──────────────┘     └──────────────┘     └──────────────┘     └──────────────┘
```

### 数据流向各环节详细说明

**1. 玩家操作**：玩家通过触摸、点击等方式与前端 UI 交互。例如点击"强化装备"按钮。

**2. 前端捕获**：前端监听用户操作事件，收集必要的参数（如装备 ID、是否使用保护符），按照协议格式构造请求 JSON。

**3. 发送请求**：前端将请求通过 HTTP POST 或 WebSocket 发送到网关层。请求中包含：`cmd`（接口编号）、`seq`（序列号，用于防止重放攻击）、`data`（业务参数）、`sign`（请求签名）。

**4. 服务端校验**：网关层先进行 TLS 解密和负载均衡分发。业务服务收到请求后，依次执行：
   - Token 校验（确认玩家已登录且 Token 有效）
   - 签名校验（确认请求未被篡改）
   - 序列号校验（防止重放攻击）
   - 参数校验（确认参数类型和范围合法）
   - 业务规则校验（如资源是否充足、状态是否允许）

**5. 逻辑运算**：校验通过后，进入核心业务逻辑。例如装备强化：
   - 根据装备当前等级和品质计算强化成功率
   - 随机数判定是否成功
   - 成功则计算强化后属性增量，更新装备表
   - 失败则处理降级或保护逻辑
   - 记录强化日志

**6. 数据存储**：将运算结果持久化到数据库，同时更新 Redis 缓存以加速后续查询。

**7. 结果封装**：将运算结果（成功/失败、新属性值、奖励列表等）封装为统一响应格式。

**8. 推送结果**：通过 HTTP 响应返回结果给客户端。对于属性变化、境界突破等需要客户端及时更新的事件，额外通过 WebSocket 推送 `push_cmd` 通知。

**9. 前端展示**：前端解析响应数据，调用对应的 UI 更新函数，将最新的游戏状态展示给玩家。

## 1.7 关键技术选型

### 1.7.1 游戏服务端

| 技术 | 选择理由 |
|:---|:---|
| **Go** | 高并发原生支持（goroutine）、简洁的并发模型、优秀的内存管理、高性能编译后二进制部署、学习曲线低，适合游戏服的高并发短连接场景 |
| **Java (可选)** | 成熟的生态体系（Spring Cloud）、丰富的中间件支持、稳定的服务治理能力，适合业务逻辑复杂的场景 |

### 1.7.2 通信协议

| 协议 | 用途 | 说明 |
|:---|:---|:---|
| **WebSocket** | 实时双向通信 | 用于战斗、挂机、属性变化推送、服务器主动通知等场景。基于 TLS 加密传输。 |
| **HTTPS** | 查询类请求 | 用于登录、角色信息拉取、排行榜查询、商店刷新等一次性查询场景。无状态，便于 CDN 加速和网关缓存。 |
| **JSON** | 数据格式 | 易于调试和日志分析，跨语言友好，客户端解析成本低。对于性能敏感的核心战斗流程，可在客户端/服务端协商后切换为二进制协议（Protobuf）。 |

### 1.7.3 数据存储

| 组件 | 选型 | 用途 |
|:---|:---|:---|
| **Redis Cluster** | 缓存与会话 | 玩家登录 Token 存储、会话数据、在线状态、排行榜数据（Sorted Set）、热点数据缓存（Redis 命中率 > 95%）、分布式锁 |

## 1.8 安全设计

安全是本游戏架构的重中之重。由于客户端完全不可信，所有安全防护机制均以服务端为核心构建。

### 1.8.1 请求签名防篡改

每个从客户端发出的请求都必须携带签名字段，服务端独立验证：

```
sign = MD5(token + seq + data + secret_key)
```

- **token**：玩家登录后下发的会话令牌
- **seq**：递增的序列号，用于防止重放攻击（服务端记录每个玩家的最新 seq，丢弃 seq 不递增的请求）
- **data**：请求的业务参数 JSON 字符串
- **secret_key**：服务端存储的私有密钥，不暴露给客户端

**验证流程**：
1. 客户端构造请求 → 计算 sign → 发送请求
2. 服务端接收请求 → 提取 token 查找对应 secret_key → 重新计算 sign → 比对
3. 签名一致则继续处理，不一致则返回错误码 `1002`（签名校验失败）

### 1.8.2 服务端校验所有客户端数据

即使客户端请求通过了签名校验，服务端仍需对数据本身进行完整校验：

- **数值范围校验**：例如境界等级必须在 1~99 之间，强化等级不能超过上限
- **资源充足性校验**：所有消耗操作前，查询数据库确认玩家实际拥有资源
- **状态前置条件校验**：例如装备强化需在背包已解锁该功能后开放
- **互斥状态校验**：例如战斗中不允许进行其他操作
- **数据一致性校验**：例如装备所属角色 ID 必须与请求者匹配

### 1.8.3 关键操作限频

对于可能影响经济系统平衡的关键操作，施加严格的频率限制：

| 操作类型 | 限制策略 | 惩罚措施 |
|:---|:---|:---|
| 装备强化 | 每分钟 N 次 | 超出返回 `4001` 错误码 |
| 挑战关卡 | 每分钟 N 次 | 超出返回 `4001` 错误码 |
| 挂机开始/结束 | 每分钟 N 次 | 超出返回 `4001` 错误码 |
| 登录验证 | 每分钟 N 次 | 超出触发账号风控 |
| 资源购买 | 每分钟 N 次 | 超出返回 `4001` 错误码 |

限频机制基于 Redis 的滑动窗口算法实现，支持精确到毫秒的时间窗口统计。

### 1.8.4 GM 指令白名单

- GM（游戏管理员）指令通过独立的 GM 接口和权限体系执行
- 每个 GM 指令必须显式声明所需权限等级
- 所有 GM 操作记录详细日志（操作者、时间、目标账号、操作内容、参数），不可删除
- GM 接口与普通玩家接口完全分离，独立部署和认证

### 1.8.5 额外安全措施

| 措施 | 说明 |
|:---|:---|
| **HTTPS 强制** | 所有 HTTP 通信强制使用 TLS 1.3 加密，防止中间人攻击 |
| **IP 限流** | 单 IP 维度的请求频率限制，防止 DDoS 和暴力破解 |
| **异常行为检测** | 同一账号短时间内大量请求、异常数值波动等触发风控告警 |
| **数据加密存储** | 敏感字段（如充值信息）在数据库中加密存储 |
| **审计日志** | 所有关键操作（充值、交易、GM 操作）写入不可篡改的审计日志表 |
