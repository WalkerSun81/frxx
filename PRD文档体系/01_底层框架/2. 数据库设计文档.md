## 2.1 ER 图（实体关系描述）

以下是核心业务实体的关系模型，采用文字描述方式呈现实体及其关联关系。

### 实体关系总览

```
Player (玩家)
    │
    └── 1 : N ── Role (角色)
                 │
                 ├── 1 : 1 ── RoleAttribute (角色属性)
                 │
                 ├── 1 : 1 ── ElementRoot (灵根)
                 │
                 ├── 1 : N ── Talent (命格天赋)
                 │       (每角色最多5个槽位)
                 │
                 ├── 1 : N ── Equipment (装备)
                 │       (角色身上穿戴的装备，一对一关系 via slot_type)
                 │
                 ├── 1 : N ── Gem (宝石)
                 │       (镶嵌在装备上或背包中的宝石)
                 │
                 ├── 1 : N ── MapProgress (地图进度)
                 │
                 ├── 1 : N ── DailyTask (每日任务)
                 │
                 ├── 1 : N ── Mail (邮件)
                 │
                 ├── 1 : 1 ── Resource (资源汇总)
                 │       (灵石、铜板、各类道具数量)
                 │
                 ├── 1 : N ── AfkRecord (挂机记录)
                 │
                 ├── 1 : N ── LifeSkill (生活技能)

Equipment (装备详情)
    │
    └── 1 : 1 ── Role (通过 role_id + slot_type 关联)
            (每种槽位同一时间只穿一件装备)

Gem (宝石)
    │
    └── 1 : 1 ── Equipment (通过嵌入口关联)
            (每个装备最多5个宝石孔)
```

### 实体详细说明

| 实体 | 英文名 | 说明 | 关联关系 |
|:---|:---|:---|:---|
| **玩家** | Player | 平台账号层面的信息，一个平台账号对应一个玩家记录 | 1 : N 角色 |
| **角色** | Role | 游戏角色，承载所有游戏数据的主实体 | N : 1 玩家 |
| **角色属性** | RoleAttribute | 角色的战斗属性（攻防血等），与角色一对一 | 1 : 1 角色 |
| **灵根** | ElementRoot | 角色的五行灵根属性，影响修炼效率和技能伤害 | 1 : 1 角色 |
| **命格天赋** | Talent | 角色的命格槽位和天赋效果，共5个槽位 | 1 : N 角色 |
| **装备** | Equipment | 角色穿戴的装备，每角色最多同时穿戴11件（含3个法宝位） | N : 1 角色 |
| **宝石** | Gem | 镶嵌在装备孔或背包中的宝石 | N : 1 角色/装备 |
| **地图进度** | MapProgress | 角色在各地图的通关进度 | 1 : N 角色 |
| **每日任务** | DailyTask | 角色的每日任务进度 | 1 : N 角色 |
| **邮件** | Mail | 系统邮件和玩家邮件 | 1 : N 角色 |
| **资源** | Resource | 角色的灵石、铜板、道具数量汇总 | 1 : 1 角色 |
| **挂机记录** | AfkRecord | 角色挂机的收益记录，按月分区 | 1 : N 角色 |
| **生活技能** | LifeSkill | 角色的生活技能（采集、制作等） | 1 : N 角色 |

## 2.2 核心表结构设计

以下为各核心数据表的详细字段定义。

### 2.2.1 玩家基础表 (t_player)

存储玩家账号层面的信息。一个平台账号对应一条玩家记录。

| 字段 | 类型 | 是否为空 | 默认值 | 说明 |
|:---|:---|:---:|:---:|:---|
| id | BIGINT | NOT NULL | | 主键，自增 |
| openid | VARCHAR(64) | NOT NULL | | 平台唯一标识（如 TapTap OpenID） |
| platform | TINYINT | NOT NULL | 1 | 平台类型：1=TapTap, 2=微信, 3=游客 |
| nickname | VARCHAR(32) | NULL | | 玩家昵称 |
| avatar_url | VARCHAR(255) | NULL | | 头像 URL |
| create_time | DATETIME | NOT NULL | CURRENT_TIMESTAMP | 账号创建时间 |
| last_login_time | DATETIME | NULL | | 最后登录时间 |
| last_login_ip | VARCHAR(45) | NULL | | 最后登录 IP（支持 IPv6） |
| total_recharge | DECIMAL(10,2) | NOT NULL | 0.00 | 累计充值金额（单位：元） |
| vip_level | TINYINT | NOT NULL | 0 | VIP 等级（0~15） |
| ban_status | TINYINT | NOT NULL | 0 | 封号状态：0=正常, 1=封禁中 |
| ban_reason | VARCHAR(255) | NULL | | 封号原因 |
| ban_end_time | DATETIME | NULL | | 封号结束时间（NULL 表示永久） |

**说明**：本表数据量预计增长较慢（每个玩家一条），但 `total_recharge` 和 `vip_level` 会被高频查询，建议在 `vip_level` 上建立索引。

### 2.2.3 角色属性表 (t_role_attribute)

存储角色的战斗属性，与角色表一一对应。

| 字段 | 类型 | 是否为空 | 默认值 | 说明 |
|:---|:---|:---:|:---:|:---|
| role_id | BIGINT | NOT NULL | | 角色 ID（主键关联）|
| strength | INT | NOT NULL | 0 | 力量（影响物理攻击）|
| vitality | INT | NOT NULL | 0 | 体魄（影响气血上限和物理防御）|
| agility | INT | NOT NULL | 0 | 敏捷（影响出手速度和闪避率）|
| intelligence | INT | NOT NULL | 0 | 灵力（影响法术攻击和魔法上限）|
| accuracy | INT | NOT NULL | 0 | 精准（影响命中率）|
| sense | INT | NOT NULL | 0 | 神识（影响暴击率和韧性）|
| wisdom | INT | NOT NULL | 0 | 悟性（影响境界突破成功率）|
| luck | INT | NOT NULL | 0 | 气运（影响掉落概率和暴击伤害）|
| hp | BIGINT | NOT NULL | 0 | 气血上限 |
| current_hp | BIGINT | NOT NULL | 0 | 当前气血 |
| mp | INT | NOT NULL | 0 | 魔法上限 |
| current_mp | INT | NOT NULL | 0 | 当前魔法 |
| physical_attack | BIGINT | NOT NULL | 0 | 物理攻击（计算后的最终值）|
| physical_defense | BIGINT | NOT NULL | 0 | 物理防御 |
| magical_attack | BIGINT | NOT NULL | 0 | 法术攻击 |
| magical_defense | BIGINT | NOT NULL | 0 | 法术防御 |
| crit_rate | DECIMAL(5,3) | NOT NULL | 0.000 | 暴击率（0.000~1.000）|
| crit_damage | DECIMAL(5,3) | NOT NULL | 1.500 | 暴击伤害倍率（默认1.5）|
| dodge_rate | DECIMAL(5,3) | NOT NULL | 0.000 | 闪避率 |
| hit_rate | DECIMAL(5,3) | NOT NULL | 1.000 | 命中率 |
| attack_speed | DECIMAL(5,2) | NOT NULL | 1.00 | 攻击速度倍率 |
| damage_increase | DECIMAL(5,3) | NOT NULL | 0.000 | 伤害增加率 |
| damage_reduce | DECIMAL(5,3) | NOT NULL | 0.000 | 伤害减免率 |
| cultivation_speed | DECIMAL(5,3) | NOT NULL | 1.000 | 修炼速度倍率（受灵根影响）|

**属性计算公式**（在服务端计算并存储最终值）：

```
physical_attack = strength * 基础系数 + 装备加成 + 宝石加成 + 命格加成
hp = vitality * 100 * (1 + 境界加成) + 装备加成 + 宝石加成
crit_rate = (sense / (sense + 1000)) * 基础倍率 + 装备加成 + 命格加成
```

### 2.2.4 灵根表 (t_element_root)

存储角色的五行灵根属性。灵根影响修炼效率和对应属性技能的伤害加成。

| 字段 | 类型 | 是否为空 | 默认值 | 说明 |
|:---|:---|:---:|:---:|:---|
| role_id | BIGINT | NOT NULL | | 角色 ID（主键关联）|
| metal | INT | NOT NULL | 0 | 金灵根值（0~100）|
| wood | INT | NOT NULL | 0 | 木灵根值 |
| water | INT | NOT NULL | 0 | 水灵根值 |
| fire | INT | NOT NULL | 0 | 火灵根值 |
| earth | INT | NOT NULL | 0 | 土灵根值 |
| main_element | TINYINT | NOT NULL | 0 | 主灵根类型：1=金, 2=木, 3=水, 4=火, 5=土, 0=未确认 |
| cultivate_multiplier | DECIMAL(4,2) | NOT NULL | 1.00 | 修炼加成倍率：灵根越均衡倍率越高，最高 2.0 |
| confirmed | TINYINT | NOT NULL | 0 | 是否已确认：0=未确认(可重随), 1=已确认 |
| update_time | DATETIME | NOT NULL | CURRENT_TIMESTAMP | 更新时间 |

**灵根规则说明**：

- 初始随机灵根：随机生成 5 个 0~100 的值，总和为 100
- 五行灵根对应五行相生相克：木生火、火生土、土生金、金生水、水生木
- 主灵根取 5 项中的最高值
- 修炼加成倍率计算：灵根方差越小（越均衡），修炼速度越快
  - 五行完全均衡（各20）：cultivate_multiplier = 2.0
  - 单灵根突出（某项>80）：cultivate_multiplier = 1.2~1.5
  - 公式：`multiplier = 1.0 + (1.0 - 标准差/50) * 1.0`，范围 [1.0, 2.0]

### 2.2.5 命格天赋表 (t_talent)

存储角色的命格天赋。角色最多拥有 5 个命格槽位（槽位可随境界解锁）。

| 字段 | 类型 | 是否为空 | 默认值 | 说明 |
|:---|:---|:---:|:---:|:---|
| id | BIGINT | NOT NULL | | 主键，自增 |
| role_id | BIGINT | NOT NULL | | 角色 ID |
| slot_index | TINYINT | NOT NULL | | 槽位编号（0~4），0=命格主位，1~4=副位 |
| slot_locked | TINYINT | NOT NULL | 0 | 槽位是否已解锁：0=锁定, 1=解锁 |
| talent_type | TINYINT | NOT NULL | 0 | 天赋类型：1=力量, 2=体魄, 3=敏捷, 4=灵力, 5=精准, 6=神识, 7=悟性, 8=气运 |
| quality | TINYINT | NOT NULL | 1 | 天赋品质：1=凡品, 2=良品, 3=上品, 4=极品, 5=神品 |
| value | INT | NOT NULL | 0 | 天赋属性加成值 |
| active | TINYINT | NOT NULL | 0 | 是否激活：0=未激活, 1=已激活 |
| create_time | DATETIME | NOT NULL | CURRENT_TIMESTAMP | 获得时间 |
| update_time | DATETIME | NOT NULL | CURRENT_TIMESTAMP | 更新时间 |

**命格类型对照表**：

| talent_type | 天赋名 | 属性加成 |
|:---:|:---|:---|
| 1 | 力命 | +力量 |
| 2 | 体命 | +体魄 |
| 3 | 速命 | +敏捷 |
| 4 | 灵命 | +灵力 |
| 5 | 准命 | +精准 |
| 6 | 神命 | +神识 |
| 7 | 悟命 | +悟性 |
| 8 | 运命 | +气运 |

**品质概率表**（随机命格时）：

| quality | 名称 | 概率 |
|:---:|:---:|:---:|
| 1 | 凡品 | 40% |
| 2 | 良品 | 35% |
| 3 | 上品 | 18% |
| 4 | 极品 | 6% |
| 5 | 神品 | 1% |

### 2.2.6 装备表 (t_equipment)

存储角色穿戴或背包中的装备信息。

| 字段 | 类型 | 是否为空 | 默认值 | 说明 |
|:---|:---|:---:|:---:|:---|

| template_id | INT | NOT NULL | 0 | 装备模板 ID（对应配置表）|
| quality | TINYINT | NOT NULL | 1 | 品质：1=白, 2=绿, 3=蓝, 4=紫, 5=橙, 6=红 |
| level | INT | NOT NULL | 0 | 强化等级（0~+30）|
| star_level | TINYINT | NOT NULL | 0 | 星级（0~5）|
| grade | TINYINT | NOT NULL | 1 | 等阶（装备等级阶段）|
| base_attrs | JSON | NULL | | 基础属性 JSON，例如：`[{"type":1, "value":100}, {"type":2, "value":50}]` |
| extra_attrs | JSON | NULL | | 额外词条 JSON（随机生成），最多4条 |
| gem_holes | JSON | NULL | | 宝石孔状态 JSON，例如：`[{"index":0,"state":0},{"index":1,"state":1}]`，state: 0=未开启, 1=开启 |
| enchant_level | INT | NOT NULL | 0 | 附魔等级 |
| enchant_attrs | JSON | NULL | | 附魔属性 JSON |
| is_locked | TINYINT | NOT NULL | 0 | 是否上锁：0=未锁, 1=上锁 |
| create_time | DATETIME | NOT NULL | CURRENT_TIMESTAMP | 获得时间 |
| update_time | DATETIME | NOT NULL | CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP | 更新时间 |

**base_attrs 字段格式示例**：

```json
[
  {"type": 1, "value": 150},    // type 1=力量, value 150
  {"type": 4, "value": 120}     // type 4=灵力, value 120
]
```

**extra_attrs 字段格式示例**：

```json
[
  {"type": "crit_rate", "value": 0.03, "quality": 2},
  {"type": "hp", "value": 500, "quality": 3}
]
```

### 2.2.7 宝石表 (t_gem)

存储宝石信息。宝石可镶嵌在装备孔中或存放在背包中。

| 字段 | 类型 | 是否为空 | 默认值 | 说明 |
|:---|:---|:---:|:---:|:---|
| id | BIGINT | NOT NULL | | 主键，自增 |
| owner_id | BIGINT | NOT NULL | 0 | 拥有者 ID：role_id（背包）或 equipment_id（镶嵌中）|
| owner_type | TINYINT | NOT NULL | 1 | 拥有者类型：1=背包, 2=装备镶嵌, 3=交易行挂单 |
| gem_type | TINYINT | NOT NULL | 1 | 宝石类型：1=生命, 2=攻击, 3=防御, 4=暴击, 5=命中, 6=闪避 |
| grade | TINYINT | NOT NULL | 1 | 品级（1~6）：1=初级, 2=中级, 3=高级, 4=顶级, 5=绝世, 6=神级 |
| slot_index | TINYINT | NOT NULL | -1 | 镶嵌槽位索引（0~4），-1 表示未镶嵌 |
| locked | TINYINT | NOT NULL | 0 | 是否上锁（0/1）|
| create_time | DATETIME | NOT NULL | CURRENT_TIMESTAMP | 创建时间 |
| update_time | DATETIME | NOT NULL | CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP | 更新时间 |

**宝石类型对照表**：

| gem_type | 宝石名 | 主属性 |
|:---:|:---|:---|
| 1 | 血玉 | +HP |
| 2 | 锐石 | +攻击 |
| 3 | 固石 | +防御 |
| 4 | 暴玉 | +暴击率 |
| 5 | 准玉 | +命中率 |
| 6 | 避玉 | +闪避率 |

**宝石合成规则**：2 颗同类型、同品级宝石可合成为 1 颗更高品级宝石（4级以上需额外消耗凝晶石）。品级上限为 6（神级）。

## 2.3 索引设计

合理的索引设计是保障查询性能的关键。以下为各表的索引规划。

### 索引总览

| 表名 | 索引名称 | 字段 | 类型 | 唯一 | 说明 |
|:---|:---|:---|:---|:---:|:---|
| t_player | idx_openid | openid | VARCHAR(64) | YES | 按平台ID查询玩家（登录时高频查询）|
| t_player | idx_platform_openid | platform + openid | TINYINT + VARCHAR(64) | YES | 联合唯一索引，支持多平台 |
| t_player | idx_last_login | last_login_time | DATETIME | NO | 用于统计 DAU、清理僵尸账号 |
| t_role | idx_player_id | player_id | BIGINT | NO | 按玩家ID查询角色列表 |
| t_role | idx_realm_level | realm_level + total_cultivate_exp | INT + BIGINT | NO | 境界排行榜索引 |
| t_role | idx_combat_power | combat_power | BIGINT | NO | 战力排行榜索引 |
| t_role_attribute | PK | role_id | BIGINT | YES | 主键关联，无额外索引 |
| t_element_root | PK | role_id | BIGINT | YES | 主键关联 |
| t_talent | idx_role_id | role_id | BIGINT | NO | 按角色ID查询所有命格 |
| t_talent | idx_role_slot | role_id + slot_index | BIGINT + TINYINT | YES | 唯一约束：每角色每槽位唯一 |
| t_equipment | idx_role_id | role_id | BIGINT | NO | 按角色查询装备列表 |
| t_equipment | idx_role_slot | role_id + slot_type | BIGINT + TINYINT | NO | 查询角色穿戴中的装备 |
| t_gem | idx_owner | owner_id + owner_type | BIGINT + TINYINT | NO | 按拥有者查询宝石 |
| t_gem | idx_owner_slot | owner_id + slot_index | BIGINT + TINYINT | NO | 查询装备上镶嵌的宝石 |
| t_map_progress | idx_role_map | role_id + map_id | BIGINT + INT | YES | 唯一约束：每角色每地图唯一 |
| t_resource | PK | role_id | BIGINT | YES | 主键关联 |
| t_afk_record | idx_role_id | role_id | BIGINT | NO | 按角色查询挂机记录 |
| t_afk_record | idx_create_time | create_time | DATETIME | NO | 按时间查询，用于数据清理 |
| t_life_skill | idx_role_skill | role_id + skill_type | BIGINT + TINYINT | YES | 唯一约束：每角色每生活技能唯一 |
| t_bag | idx_role_template | role_id + template_id | BIGINT + INT | NO | 道具叠加查询 |
| t_mail | idx_role_unread | role_id + read_status + expire_time | BIGINT + TINYINT + DATETIME | NO | 查询未读邮件 |

### 索引设计原则

1. **高频查询字段优先索引**：登录时按 `openid` 查询玩家，按 `player_id` 查询角色，这些字段必须有索引
2. **唯一约束字段建立唯一索引**：如 `t_player.openid`、`t_equipment.role_id + slot_type` 组合
3. **联合索引遵循最左前缀原则**：如 `idx_role_skill(role_id, skill_type)` 支持按 `role_id` 查询，也支持按 `(role_id, skill_type)` 精确查询
4. **避免过度索引**：每个额外索引都会增加写入开销（Insert/Update 时需同时维护索引树），控制在 3~5 个索引/表
5. **排序字段建立索引**：排行榜查询通常按战力/境界/修为降序排序，在相应字段上建立索引可加速排序

## 2.4 数据分区策略

随着游戏运营时间增长，数据量会持续膨胀。合理的分区策略可以：

- 显著提升查询性能（只扫描相关分区）
- 方便冷数据归档（历史分区可直接迁移到归档库）
- 降低单表数据量，减少索引维护开销

### 2.4.1 挂机记录表按月份分区

`t_afk_record` 表数据量最大（每分钟可产生多条记录），采用**按月 Range 分区**：

```sql
CREATE TABLE t_afk_record (
    id BIGINT NOT NULL AUTO_INCREMENT,
    role_id BIGINT NOT NULL,
    ...
    PRIMARY KEY (id, create_time),  -- 注意：分区键必须在主键中
    INDEX idx_role_id (role_id),
    INDEX idx_create_time (create_time)
)
PARTITION BY RANGE (TO_DAYS(create_time)) (
    PARTITION p202605 VALUES LESS THAN (TO_DAYS('2026-06-01')),
    PARTITION p202606 VALUES LESS THAN (TO_DAYS('2026-07-01')),
    PARTITION p202607 VALUES LESS THAN (TO_DAYS('2026-08-01')),
    PARTITION p_future VALUES LESS THAN MAXVALUE
);
```

**运维策略**：

- 每月初自动创建下月分区（如5月底创建 `p202606`）
- 每 3 个月将历史分区（6个月前的数据）归档到冷数据数据库
- 归档前确认归档任务的完整性和数据校验

### 2.4.2 邮件表按过期时间分区

`t_mail` 表中大量邮件在过期后成为死数据，采用**按月份分区**加速清理：

```sql
CREATE TABLE t_mail (
    id BIGINT NOT NULL AUTO_INCREMENT,
    role_id BIGINT NOT NULL,
    expire_time DATETIME NOT NULL,
    ...
    PRIMARY KEY (id, expire_time),
    INDEX idx_role_unread (role_id, read_status, expire_time)
)
PARTITION BY RANGE (TO_DAYS(expire_time)) (
    PARTITION p_expired VALUES LESS THAN (TO_DAYS('2026-05-01')),
    PARTITION p_may VALUES LESS THAN (TO_DAYS('2026-06-01')),
    PARTITION p_future VALUES LESS THAN MAXVALUE
);
```

### 2.4.3 排行榜数据使用 Redis Sorted Set

排行榜数据（战力榜、境界榜等）不存储在 MySQL 中，而是使用 Redis Sorted Set：

```
Key: ranking:combat_power
Score: combat_power 数值
Member: role_id
```

- 查询 Top N：`ZREVRANGE ranking:combat_power 0 99 WITHSCORES`
- 查询玩家排名：`ZREVRANK ranking:combat_power {role_id}`
- 更新：`ZADD ranking:combat_power {combat_power} {role_id}`

Sorted Set 的时间复杂度为 O(log(N))，10 万玩家数据的排名查询可在毫秒级完成。

### 2.4.4 冷数据归档策略

| 数据类型 | 热数据保留周期 | 归档位置 | 说明 |
|:---|:---:|:---|:---|
| 挂机记录 | 3 个月 | 归档库 t_afk_record_history | 按月份分区后归档 |
| 邮件 | 7 天 | 自动删除 | 过期后由定时任务清理 |
| 登录日志 | 1 个月 | 归档库 t_login_log_history | 用于统计分析 |
| 操作日志 | 6 个月 | 归档库 t_operation_log_history | 审计用途 |
| 战斗日志 | 1 个月 | 归档库 t_battle_log_history | 用于数据分析 |
