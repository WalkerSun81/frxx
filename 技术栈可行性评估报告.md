# 《凡人修仙录》技术栈可行性评估报告

> 评估日期：2026-05-12  
> 评估对象：纯前端方案 vs 轻量后端方案

---

## 一、评估背景

根据文档更新要求，前端技术栈已从Cocos Creator/Unity改为**纯HTML5**，需要重新评估：
1. 纯前端方案（HTML + 浏览器本地存储）的可行性
2. 轻量后端方案（Node.js + SQLite）的必要性
3. 两种方案的优劣势对比

---

## 二、方案一：纯前端方案（HTML + LocalStorage）

### 2.1 架构设计

```
┌─────────────────────────────────────────┐
│           浏览器环境                     │
│  ┌─────────────────────────────────┐   │
│  │      单个HTML文件                │   │
│  │  ┌─────────┐  ┌─────────────┐  │   │
│  │  │ 游戏逻辑 │  │ UI渲染/交互  │  │   │
│  │  │ (JS)    │  │ (HTML/CSS)  │  │   │
│  │  └────┬────┘  └─────────────┘  │   │
│  │       │                        │   │
│  │  ┌────▼────────────────────┐  │   │
│  │  │    LocalStorage         │  │   │
│  │  │  (5~10MB存储上限)        │  │   │
│  │  │  - 玩家存档数据          │  │   │
│  │  │  - 游戏配置表            │  │   │
│  │  └─────────────────────────┘  │   │
│  └─────────────────────────────────┘   │
└─────────────────────────────────────────┘
```

### 2.2 可行性分析

| 维度 | 可行性 | 说明 |
|:---|:---:|:---|
| **离线运行** | ✅ 完全支持 | 双击HTML文件即可运行，无需网络 |
| **数据存储** | ⚠️ 受限 | LocalStorage 5~10MB上限，IndexedDB可扩展至50MB+ |
| **数值计算** | ❌ 不可信 | 所有计算在客户端，易被修改（F12控制台） |
| **跨设备同步** | ❌ 不支持 | 数据绑定单设备浏览器 |
| **反作弊** | ❌ 无法实现 | 玩家可随意修改存档数据 |
| **多人在线** | ❌ 不支持 | 无服务器无法实现排行榜/PVP等 |

### 2.3 核心问题清单

#### 问题1：存档篡改（致命）

```javascript
// 玩家可在控制台执行：
localStorage.setItem('xiuxian_save', JSON.stringify({
  realm_level: 36,  // 直接改为渡劫圆满
  gold: 999999999,  // 无限灵石
  // ... 任意修改
}));
```

**影响**：游戏经济系统完全崩溃，失去游戏性。

#### 问题2：存储容量限制

| 数据类型 | 预估大小 | 是否可行 |
|:---|:---:|:---:|
| 角色基础数据 | ~5KB | ✅ |
| 装备数据(100件) | ~50KB | ✅ |
| 挂机记录(30天) | ~200KB | ✅ |
| 游戏配置表 | ~500KB | ✅ |
| 图片资源(base64) | ~2MB+ | ⚠️ 需CDN |
| **总计** | **~3MB** | ✅ 在限制内 |

#### 问题3：数值计算一致性

PRD中所有伤害公式、突破成功率、掉落概率都依赖服务端计算。纯前端方案下：
- 玩家可修改随机数种子，使每次突破100%成功
- 可修改掉落率，每次战斗必掉橙装
- 可修改战斗公式，一击必杀任何Boss

### 2.4 适用场景

✅ **适合**：
- 单机演示版
- 个人本地娱乐
- 无需反作弊的纯体验版

❌ **不适合**：
- 正式上线运营
- 有排行榜/PVP等社交功能
- 需要数据持久化和跨设备同步

## 三、方案二：轻量后端方案（Node.js + SQLite）

### 3.1 架构设计

```
┌─────────────────────────────────────────────────────────┐
│                    客户端层 (HTML5)                      │
│                                                         │
│    单个HTML文件 ──► 纯展示 + 交互捕获                    │
│    (浏览器直接打开，无需构建)                             │
│                      │                                  │
│                      │ HTTP / WebSocket                 │
│                      ▼                                  │
├─────────────────────────────────────────────────────────┤
│                   后端控制台 (Node.js)                   │
│                                                         │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐    │
│  │  Express    │  │  游戏逻辑    │  │  WebSocket  │    │
│  │  REST API   │  │  数值计算    │  │  实时推送   │    │
│  └──────┬──────┘  └──────┬──────┘  └──────┬──────┘    │
│         │                │                │           │
│  ┌──────▼────────────────▼────────────────▼──────┐    │
│  │              SQLite 数据库                      │    │
│  │  - t_player (玩家账号)                          │    │
│  │  - t_role (角色数据)                            │    │
│  │  - t_equipment (装备)                           │    │
│  │  - t_afk_record (挂机记录)                      │    │
│  └───────────────────────────────────────────────┘    │
│                                                         │
│  ┌───────────────────────────────────────────────┐    │
│  │           内置Web管理面板                       │    │
│  │  - 玩家管理 / 数据统计 / 配置热更新              │    │
│  └───────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────┘
```

### 3.2 技术选型详情

| 组件 | 选型 | 理由 |
|:---|:---|:---|
| **运行时** | Node.js 18+ | LTS版本，生态成熟，单文件可执行 |
| **Web框架** | Express 4.x | 轻量、文档丰富、中间件生态完善 |
| **数据库** | SQLite3 | 零配置、单文件存储、足够支撑10万玩家 |
| **实时通信** | ws (WebSocket) | 原生实现，无额外依赖 |
| **进程管理** | PM2 (生产) | 自动重启、日志管理、集群模式 |
| **部署** | Docker单容器 | 一键启动，降低运维门槛 |

### 3.3 可行性分析

| 维度 | 可行性 | 说明 |
|:---|:---:|:---|
| **离线运行** | ❌ 不支持 | 需要启动后端服务 |
| **数据安全** | ✅ 高 | 所有数据在服务端，客户端无法篡改 |
| **数值计算** | ✅ 可信 | 伤害/掉落/突破等计算在服务端完成 |
| **跨设备同步** | ✅ 支持 | 通过账号系统实现多设备数据同步 |
| **反作弊** | ✅ 可实现 | 服务端校验所有请求，拒绝非法操作 |
| **部署复杂度** | ⚠️ 中等 | 需安装Node.js，但可打包为单可执行文件 |

### 3.4 部署方案对比

#### 方案A：开发环境（零配置）

```bash
# 1. 安装Node.js（官网下载安装包）
# 2. 启动后端
cd backend
npm install  # 首次
npm start    # 启动服务

# 3. 打开前端
double-click frontend/index.html  # 浏览器打开
```

**适用**：本地开发、个人测试

#### 方案B：生产环境（Docker）

```dockerfile
# Dockerfile
FROM node:18-alpine
WORKDIR /app
COPY package*.json ./
RUN npm install --production
COPY . .
EXPOSE 3000
CMD ["node", "server.js"]
```

```bash
# 构建并运行
docker build -t xiuxian-game .
docker run -p 3000:3000 -v $(pwd)/data:/app/data xiuxian-game
```

**适用**：小范围测试、私服运营

#### 方案C：单可执行文件（pkg）

```bash
# 使用pkg打包为单文件
npm install -g pkg
pkg server.js --targets node18-win-x64

# 生成 xiuxian-server.exe，双击即可运行
```

**适用**：给非技术人员部署、极简场景

### 3.5 性能评估

| 指标 | 预估 | 说明 |
|:---|:---:|:---|
| **并发连接** | 1000+ | Node.js事件循环，非阻塞IO |
| **数据库容量** | 10万玩家 | SQLite单文件上限281TB，实际受磁盘限制 |
| **内存占用** | ~200MB | Node.js + SQLite + 缓存 |
| **启动时间** | <3秒 | 轻量服务，快速启动 |
| **API延迟** | <50ms | 本地/局域网环境 |

### 3.6 运维成本

| 任务 | 复杂度 | 频率 |
|:---|:---:|:---:|
| 服务启动 | 低 | 每次开机 |
| 数据备份 | 低 | 定时复制data目录 |
| 日志查看 | 低 | 实时/按需 |
| 配置更新 | 低 | 热更新，无需重启 |
| 版本升级 | 中 | 替换文件，保留data目录 |

## 五、推荐方案

### 5.1 最终推荐：轻量后端方案（Node.js + SQLite）

**推荐理由**：

1. **数据安全是底线**：修仙游戏的核心是数值成长，如果玩家可以随意修改存档，游戏失去意义
2. **PRD设计依赖后端**：境界突破成功率、装备掉落、战斗伤害等核心玩法都设计为服务端计算
3. **未来扩展性**：即使当前只需要单机，保留后端架构也为后续联机功能（排行榜、PVP）预留空间
4. **部署成本可接受**：通过pkg打包为单可执行文件，非技术人员也能一键启动

### 5.2 折中方案：双模式支持

如果坚持"双击HTML即可运行"的极致体验，可采用**双模式架构**：

```
┌─────────────────────────────────────────┐
│              前端 (HTML5)                │
│  ┌─────────────────────────────────┐   │
│  │      模式检测                    │   │
│  │  if (后端可用) ──► 连接服务端    │   │
│  │  else ──► 本地模拟模式           │   │
│  └─────────────────────────────────┘   │
└─────────────────────────────────────────┘
```

**本地模拟模式**：
- 使用LocalStorage存储数据
- 所有计算在本地执行（接受被篡改的风险）
- 显示"单机模式 - 数据仅本地存储"提示

**服务端模式**：
- 正常连接Node.js后端
- 数据安全，支持同步

### 5.3 实施建议

| 阶段 | 方案 | 说明 |
|:---|:---|:---|
| **MVP演示** | 纯前端 | 快速验证核心玩法，接受数据不安全 |
| **内部测试** | 轻量后端 | 部署Node.js服务，收集真实数据 |
| **小规模上线** | 轻量后端 + Docker | 容器化部署，便于运维 |
| **正式运营** | 轻量后端 → 标准后端 | 迁移至MySQL/Redis集群架构 |

## 六、结论

| 问题 | 结论 |
|:---|:---|
| 纯前端方案可行吗？ | ⚠️ 技术上可行，但**不适合正式上线**，仅适合演示/个人娱乐 |
| 需要后端控制台吗？ | ✅ **需要**，数据安全和反作弊是游戏的基础要求 |
| 推荐技术栈 | **Node.js + Express + SQLite**，轻量且足够支撑初期运营 |
| 能否实现"双击运行"？ | ✅ 可以，通过pkg打包为单可执行文件，非技术人员也能启动 |
